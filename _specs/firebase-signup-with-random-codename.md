# Spec for firebase-signup-with-random-codename

branch: claude/feature/firebase-signup-with-random-codename

## Summary

Connect the existing signup form to Firebase Authentication using the Firebase Web SDK. On successful account creation, generate a random PascalCase codename for the user (combining one word from each of three distinct word sets), set it as the user's `displayName` in Firebase Auth, and create a corresponding document in the Firestore `users` collection containing only the user's `id` and `codename` (no email).

## Functional requirements

- The signup form in `app/(public)/signup/` must call Firebase Auth's `createUserWithEmailAndPassword` on submission, using the auth instance exported from `lib/firebase/config.ts`.
- A `lib/firebase/config.ts` file must export the initialized Firebase app, auth instance, and Firestore instance, with all credentials read from environment variables (never hardcoded).
- After successful account creation, a random codename must be generated by picking one word from each of three separate word lists and concatenating them in PascalCase (e.g., `SwiftCrimsonFox`).
- Each of the three word lists must contain unique words relevant to the office-heist theme of the application.
- The generated codename must be set as the Firebase Auth user's `displayName` using `updateProfile`.
- A document must be written to the Firestore `users` collection with the following fields only:
  - `id`: the Firebase Auth user's UID
  - `codename`: the generated PascalCase codename
  - The user's email must NOT be stored.
- The codename generation logic must live in a dedicated utility (e.g., `lib/utils/generateCodename.ts`) to keep it testable and reusable.
- Only the Firebase Web SDK (modular API) must be used — no Admin SDK.
- On signup error (e.g., email already in use, weak password), a meaningful error message must be surfaced to the user in the form.
- On success, the user should be redirected to the authenticated area of the app.

## Possible Edge Cases

- Firebase project not configured: app crashes or silently fails if env vars are missing.
- Email already registered: Firebase returns `auth/email-already-in-use` — must be caught and shown.
- Weak password: Firebase returns `auth/weak-password` — must be caught and shown.
- Codename collision in Firestore: two users could theoretically generate the same codename; this is acceptable for now given the word list size.
- Network failure mid-signup: user account may be created in Auth but Firestore write could fail — the user is still authenticated but lacks a `users` document.
- The word lists must each contain enough unique entries to produce a reasonably large codename space.

## Acceptance Edge Cases

- Submitting the form with a valid email and password creates a Firebase Auth user.
- The created user's `displayName` is not null and matches a valid PascalCase codename pattern.
- The `users` Firestore collection gains a new document with `id` and `codename` matching the new user — no `email` field present.
- Re-submitting with the same email shows an error and does not create a duplicate user.
- Submitting with a password shorter than 6 characters shows a password error.

## Acceptance Criteria

- `lib/firebase/config.ts` exports `app`, `auth`, and `db` using only environment variables for configuration.
- `lib/utils/generateCodename.ts` exports a function that returns a PascalCase string built from three word lists.
- The signup form calls Firebase Auth on submit and handles loading and error states.
- `updateProfile` is called with the generated codename as `displayName` after user creation.
- A Firestore document is written to `users/{uid}` with `id` and `codename` fields only.
- All Firebase credentials are stored in `.env.local` and never appear in source files.
- Unit tests exist for the codename generator covering: output format (PascalCase), word origin (one from each list), and deterministic structure.

## Open Criteria

- Whether to handle the edge case where Firestore write fails post-auth (e.g., retry logic or cleanup).

## Testing Guidelines

Create a test file(s) in the ./tests folder for the new feature, and create meaningful tests for the following cases, without going too heavy:

- `generateCodename` returns a string in PascalCase format.
- `generateCodename` output always consists of exactly three words joined together (verifiable by checking known word lists).
- `generateCodename` picks from all three word lists (each call includes a word from each list).
- Each word list contains only unique entries (no duplicates).
